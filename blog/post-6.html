Building at Scale: Accessible UI Components for a Global Insurance Platform

March 19, 2026 6 min read

Working on Unum Ireland's platform means building UI that is used by employers, employees, and HR administrators across multiple countries. The surface area is large, the requirements are strict, and accessibility is non-negotiable — in a regulated industry, it's a legal requirement, not a nice-to-have.

What Scale Actually Demands

Scale doesn't just mean more users — it means more edge cases, more consuming teams, and more things that can go wrong silently. A component that renders correctly for a two-word label may break completely with a 60-character name and a long employer reference code. Every component I build gets tested against extremes: empty states, truncated content, RTL text, high-contrast mode, and screen magnification up to 400%.

The discipline this creates is valuable: you stop thinking about the happy path and start designing for the full input space. That habit carries over to every other project.

Accessibility as an Engineering Constraint

WCAG 2.1 AA compliance means the component library has specific, testable requirements — not aspirations. Some of the constraints that shape everyday decisions:

  • Colour contrast ratios are checked programmatically on every token change
  • Focus order must follow the visual DOM order; reordering with CSS alone is forbidden
  • Every interactive control has an accessible name — either visible label text or aria-label
  • Form validation errors are associated with their inputs via aria-describedby and announced via aria-live
  • Complex widgets (date pickers, data tables, modals) follow the ARIA Authoring Practices Guide patterns exactly

The hardest accessibility work is in the composite components. A data table with sortable columns, row selection, and inline editing has to manage focus precisely — after a row action, focus must return to a sensible, predictable location. Getting this right requires testing with NVDA, JAWS, and VoiceOver across browsers, not just checking axe audit scores.

TypeScript at the Component Boundary

In a shared component library, TypeScript is load-bearing. Prop types are the contract between the component and every team that uses it. When the contract is well-defined, consuming teams get autocomplete, error messages before runtime, and confidence that a component update won't silently break their page.

The patterns I reach for most often: 

// Discriminated unions for variant components
type AlertProps =
  | { variant: 'info'; message: string }
  | { variant: 'error'; message: string; onRetry?: () => void }
  | { variant: 'success'; message: string; autoDismiss?: boolean };

// Generic data components
function DataTable<T extends Record<string, unknown>>(
  props: DataTableProps<T>
): JSX.Element

What This Work Has Taught Me

Building for a platform at this scale is slower and more constrained than building a side project. The constraints are the point. Every shortcut you don't take, every edge case you handle, every screen reader announcement you get right — these compound into a system that teams trust and users rely on.

The best side effect of enterprise-scale accessibility work is that it makes everything else you build better, almost automatically.
