Building a Touch Typing Tool with Vue 3 and TypeScript

March 12, 2026 5 min read

I built touch-typing-tool to sharpen my own typing speed and accuracy. What started as a quick Vue 3 experiment turned into a lesson in input handling, real-time state management, and the surprisingly tricky UX of making feedback feel instant.

Why Vue 3 and TypeScript?

Most of my professional work lives in React, so building side projects in Vue keeps the comparison fresh and honest. Vue 3's Composition API is genuinely pleasant to work with — ref, computed, and watchEffect map cleanly onto the reactive state a typing tool needs. TypeScript catches the category of bugs that show up when you're tracking keystrokes, indices, and timing all at once.

The Keystroke Logic

The core loop is straightforward: compare each typed character against the expected character at the current cursor position, advance on a match, mark an error on a mismatch. The tricky part is deciding how errors behave. Options:

  • Block on error — user must fix before advancing (strict mode)
  • Allow through — mark incorrect characters, continue typing
  • Count errors but permit free movement

I went with block-on-error for the default mode. It's more frustrating initially, but it builds cleaner muscle memory — the goal of a typing trainer. The key insight is that the input field itself should have zero visible UI chrome. Users should feel like they're typing directly onto the text, not into a box. 

// Track per-character state
type CharState = 'pending' | 'correct' | 'incorrect';

const chars = computed(() =>
  prompt.value.split('').map((ch, i) => ({
    char: ch,
    state: getCharState(i) as CharState,
  }))
);

WPM Calculation

Words per minute is conventionally calculated as characters typed divided by 5 (the average English word length), divided by minutes elapsed. The important detail is when you start the timer: on the first keypress, not on page load. Starting the clock too early inflates the apparent difficulty; starting it too late means the first burst of typing is unmetered.

Accuracy is a separate metric: correct characters divided by total keystrokes (including backspaces in some modes). Displaying both live prevents users from gaming one at the expense of the other.

Making Feedback Feel Instant

Perceived latency between a keypress and visual feedback is the most important UX property of a typing tool. Anything over about 50ms feels sluggish. The solution is keeping all state in reactive primitives that update synchronously on each keyboard event — no async operations, no debouncing on the hot path, no DOM queries inside the event handler.

The final touch was using CSS custom properties to drive character colour. A single class toggle on each character span drives the correct/incorrect/cursor state without triggering layout, keeping the render budget well within frame even on long prompts.
