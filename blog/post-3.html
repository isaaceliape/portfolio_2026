mark-board: File-Based Kanban with AI-Powered User Stories

March 14, 2026 6 min read

Most Kanban tools store tasks in a database you can't easily inspect, diff, or version-control. mark-board flips that: tasks live as markdown files in folders, which means your entire board is just a directory tree you can grep, commit, and back up like any other code.

The Core Idea: Board as Filesystem

The directory structure mirrors the board columns directly: 

board/
  backlog/
    add-dark-mode.md
    improve-search.md
  in-progress/
    fix-nav-bug.md
  done/
    initial-setup.md

Moving a card is a file rename. Deleting a card is removing a file. The entire board state is always visible to any tool that can read a filesystem — editors, shell scripts, CI pipelines. This is a deliberate constraint, not a limitation.

Reading and Writing with the File System Access API

On the web, the File System Access API lets the app request a directory handle from the user and then read and write files within it persistently across sessions. The board mounts a directory, reads all .md files recursively, and parses each one as a task card.

The React/TypeScript layer is deliberately thin here — a useBoard hook manages the directory handle, reads the file tree on mount, and exposes moveCard, createCard, and deleteCard operations that each map to a single filesystem action.

AI User Story Generation via OpenRouter

The AI integration is optional but useful: given a card title, OpenRouter generates a full user story in the format "As a [user], I want [action] so that [benefit]", along with acceptance criteria. This saves the blank-page friction when writing a new ticket from scratch.

OpenRouter was the right choice here because it gives access to multiple model providers through a single API. The generation prompt is simple — the quality comes from giving the model enough context about the board and the column the card is being created in. 

const story = await generateUserStory({
  title: card.title,
  column: 'backlog',
  boardContext: board.name,
});

What This Architecture Enables

Because the board is just files, you get some things for free that would require significant engineering in a traditional Kanban tool:

  • Full git history of every task change
  • Grep the entire board with standard shell tools
  • Branch per feature — switch branches, switch board state
  • Offline-first by default — no sync, no conflicts

The tradeoff is that real-time collaboration is hard — you'd need to layer a sync mechanism on top. For solo developers and small teams, the filesystem model is surprisingly complete.
