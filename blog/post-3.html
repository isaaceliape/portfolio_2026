Optimizing Performance in Modern JavaScript

March 4, 2026 5 min read

Performance isn't just about bundle size. In modern JavaScript applications, speed is a feature that requires attention at every layer—from initial load to interaction responsiveness. Here's how I approach it.

Understanding Core Web Vitals

Google's Core Web Vitals give us concrete metrics to optimize for:

  • LCP (Largest Contentful Paint): Target < 2.5s. This measures when the largest visible content element loads.
  • INP (Interaction to Next Paint): Target < 200ms. This measures how quickly the page responds to user interactions.
  • CLS (Cumulative Layout Shift): Target < 0.1. This measures visual stability—how much elements move around unexpectedly.

These aren't just numbers—they represent real user experiences. A slow LCP means users staring at blank screens. A poor INP means frustrating delays when clicking buttons. High CLS means accidental clicks on wrong elements.

Strategic Code Splitting

Not every user needs every line of JavaScript on first load. Modern bundlers make code splitting straightforward: 


// Route-based splitting
const Dashboard = lazy(() => import('./Dashboard'));

// Component-based splitting for heavy features
const HeavyChart = lazy(() => import('./HeavyChart'));

// Preload on interaction
const handleHover = () => {
  const HeavyComponent = import('./HeavyComponent');
};

The key is being intentional about what loads when. Critical path code loads immediately. Everything else waits until needed.

Lazy Loading Strategies

For more control, the Intersection Observer API lets you trigger loads based on scroll position, hover, or other custom conditions.

Bundle Size Budgets

What gets measured gets managed. Set bundle size budgets and fail builds when they exceed limits:

  • Entry point: < 170KB gzipped
  • Lazy-loaded routes: < 250KB gzipped
  • Shared chunks: Monitor growth over time

Tools like webpack-bundle-analyzer or @vitejs/plugin-legacy make it easy to visualize where your bytes go.

The Performance Mindset

Optimization is not a one-time task—it's a continuous discipline:

  • Measure before optimizing (profile first, guess never)
  • Test on real devices and slow networks
  • Monitor Core Web Vitals in production
  • Prioritize user-perceived performance over raw metrics

The fastest code is the code you don't ship. Before adding a library, ask if the platform provides what you need. Before loading an asset, ask if it's essential for the first paint.
